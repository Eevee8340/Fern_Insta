# Long-Term Memory & Workflow Manual

This manual describes the architecture, data flow, and retrieval mechanisms of the Fern AI memory system. Fern does not just "remember" text; it actively processes, categorizes, and retrieves information using a sophisticated RAG (Retrieval-Augmented Generation) pipeline.

## 1. The Memory Architecture

Fern uses **ChromaDB**, a vector database, to store information in a high-dimensional space. This allows the bot to find "semantically similar" information even if the exact words don't match.

### Core Storage Types:
- **Strict Facts (`source='strict_fact'`):** Atomic, verified statements about users (e.g., "Alice likes pizza").
- **Narrative Logs (`source='log_entry'`):** Structured summaries of past conversations generated by the `Summarizer` plugin.
- **Raw Context:** Short-term chat history (last 30 messages) held in volatile RAM.

---

## 2. The Ingestion Workflow (Data Collection)

Memory ingestion is a tiered, iterative process:

### Step 1: Rolling Context (Short-Term Summary)
Every 50 messages, the `Summarizer` (Archivist) plugin updates a **Rolling Summary**.
- It takes the previous summary and the new 50 messages to generate an updated report.
- This ensures the bot's "immediate memory" is cumulative and context-aware.

### Step 2: Immediate Fact Extraction (Medium-Term)
Every 50 messages, the updated Rolling Summary is sent to the `Cortex` for **Strict Fact Extraction**.
- This allows the bot to learn "instantly" (every 50 messages) rather than waiting for long cycles.
- Facts are tagged with specific users if their names are mentioned.

### Step 3: Conflict Resolution
When saving a fact, the `MemoryDB` checks for conflicts:
- **Refresh:** If a similar fact exists (dist < 0.15), the timestamp is updated.
- **Overwrite:** If a conflicting fact for the same user is detected (dist < 0.35), the old fact is deleted and replaced by the new one.

### Step 4: Final Archiving (Long-Term)
After 5 iterations (250 messages), the final cumulative Rolling Summary is saved to ChromaDB as a **Narrative Log** and the cycle resets.

---

## 3. The Retrieval Workflow (RAG)

### Step 1: User-Scoped Search
The user's message is vectorized. Fern queries the database for logs and facts.
- **Fact Filtering:** Retrieval is scoped to the current user (using their handle/alias) and "system" facts. This prevents "context smearing" from other users.

### Step 2: Context Purification
Before injection into the prompt, Narrative Logs are "purified."
- Metadata like `[USER_STATE]` and `[NEW_FACTS]` are stripped via regex.
- Only the `[TOPICS]` and `[KEY_EVENTS]` (the narrative thread) are passed to the LLM to keep the prompt clean.

### Step 2: Reranking
Since vector search can sometimes be "fuzzy," Fern uses a **Cross-Encoder Reranker** (`ms-marco-MiniLM-L-6-v2`). This secondary AI model looks at the user's query and the 25 candidates to determine which 10 are *truly* the most relevant.

### Step 3: Context Injection
The top 10 verified facts and relevant logs are formatted into a special system block:
```text
### CRITICAL MEMORY CONTEXT
The following facts are verified history. You MUST align your response with them:
- Alice started playing "Elden Ring".
- Alice thinks the server is on fire.
```
This block is injected into the LLM's prompt *before* it generates a response.

---

## 4. Maintenance & Evolution

- **Memory Clearing:** The `/clearmem` command wipes the *short-term* history and buffers but preserves the *long-term* vector database.
- **Lore Integration:** The `Lore` plugin periodically scans these same memory logs to extract group slang and memes, adding another layer of "cultural memory."
- **Profile Sync:** The `Profiler` plugin uses the narrative logs to update user traits and relationship statuses, ensuring the bot's social memory matches the conversation history.

## 5. Summary of Workflow

1. **Chat Message** -> 2. **Summarizer Buffer** -> 3. **Narrative Log** -> 4. **Cortex Analysis** -> 5. **Vector Storage** -> 6. **Semantic Query (on new message)** -> 7. **Reranking** -> 8. **Prompt Injection** -> 9. **Context-Aware Reply**.
